gifsicle.wasm 은 무엇인가요?
gifsicle 은 1997년부터 유지되어온 GIF 압축·편집의 표준 도구입니다. Eddie Kohler (Harvard CS 교수) 가 만든 C library 로, 거의 모든 리눅스 배포판에 기본 포함. WebAssembly 포트로 브라우저 안에서 그대로 실행됩니다. 색상수·lossy·optimize 옵션은 원본 gifsicle CLI 와 동일.
큰 파일 (50MB+) 처리?
기술적으로 가능하지만 wasm 메모리 제약 (모바일 ~500MB, 데스크톱 Chrome ~2GB) 때문에 큰 GIF 는 멈출 수 있습니다. 먼저 video/to-gif 도구로 fps·해상도를 줄여 만든 후 압축하는 게 효율적. 큰 GIF 의 원인은 보통 fps 30+·해상도 1280×720+.
왜 압축 강도 5도 원본의 50% 정도까지만 줄어드나요?
GIF 는 이미 LZW 압축된 포맷이라 추가 압축에 한계가 있습니다. 더 큰 감소를 원하면: ① fps 줄임 (10fps → 50% 감소), ② 해상도 줄임 (640 → 320 = 75% 감소), ③ 프레임 수 자체 줄임. 이 3가지는 video/to-gif 도구에서 처음부터 만들 때 적용.
모바일에서도 작동하나요?
작동합니다 (iOS Safari·Chrome·Firefox 모두 wasm 지원). 단 메모리 제약상 10MB 이하 GIF 권장. 데스크톱 Chrome 은 100MB까지 안정. 처음 사용 시 gifsicle.wasm 다운로드 (~200KB) 후 캐시되어 재사용.

자주 묻는 질문

GIF 압축 후에도 애니메이션 유지되나요?
유지됩니다. gifsicle.wasm 은 multi-frame GIF 의 모든 프레임을 보존하면서 색상수·lossy·optimize 옵션으로 용량을 줄입니다. canvas API 방식 (이미지 압축 도구) 은 첫 프레임만 남기지만, 이 도구는 GIF 전용이라 애니메이션 그대로 유지.
lossy 옵션은 무엇인가요?
픽셀 색을 약간 흔들어 LZW 압축 효율을 높이는 기법입니다. 값이 클수록 더 작아지지만 노이즈도 보이기 시작합니다. 권장 0~80 (시각 손실 거의 없음). 100~200 (SNS·메신저 충분, 약간 노이즈).
색상수를 줄이면 어떻게 되나요?
GIF 는 256색 팔레트 기반입니다. 색상수를 128·64·32 로 줄이면 용량이 작아지지만 그라데이션·부드러운 색 변화가 줄어듭니다. 단색·로고·일러스트 같은 단순 GIF 는 64색으로도 충분. 사진·실사 GIF 는 256 권장.
왜 이미지 압축 도구로는 GIF 가 안 되나요?
이미지 압축 도구는 canvas API 의 toBlob() 으로 처리하는데, canvas 는 multi-frame 미지원입니다. GIF 를 업로드하면 첫 프레임만 추출되어 JPG·PNG·WebP 로 변환됩니다. 애니메이션을 유지하려면 이 GIF 압축 도구를 사용하세요.
이미 작은 GIF 도 더 작아지나요?
한계가 있습니다. 이미 100KB 이하 작은 GIF 는 추가 압축 효과가 10% 미만일 수 있음. 큰 GIF (1MB+) 에서 가장 효과적 (30~80% 감소).
여러 GIF 한 번에 처리할 수 있나요?
현재는 1장씩 처리. 여러 장 batch 처리는 다음 update 예정. 대량 처리가 필요하면 강도 결정 후 한 장씩 빠르게 반복 처리하시면 됩니다.
