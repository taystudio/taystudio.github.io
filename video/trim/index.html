대부분 빠른 모드로 충분합니다. 시작 시각이 정확히 맞아야 할 때만 정확 모드.

왜 빠른 모드는 시작 시각이 정확히 안 맞을 때가 있나요?
H.264 같은 영상 코덱은 키프레임(I-frame)이라는 독립 프레임 사이를 다른 프레임의 차이로 압축합니다. 재인코딩 없이 자르려면 키프레임 경계에서만 잘라야 영상이 깨지지 않으며, 키프레임은 보통 1~5초 간격으로 배치됩니다. 그래서 빠른 모드는 지정한 시각보다 가까운 앞쪽 키프레임으로 시작점이 끌려갈 수 있습니다.
처리 도중 페이지를 닫아도 되나요?
처리 진행 중에 페이지를 닫으면 작업이 취소됩니다. 다운로드된 ffmpeg 라이브러리는 캐시에 남아 다음에 재사용되므로, 닫고 다시 들어와도 라이브러리는 다시 받지 않습니다.
왜 멀티 스레드(빠른 처리)는 지원 안 되나요?
GitHub Pages 같은 정적 호스팅은 Cross-Origin-Opener-Policy / Cross-Origin-Embedder-Policy 헤더를 설정할 수 없어 SharedArrayBuffer가 비활성됩니다. 정확 모드만 영향(재인코딩 시 단일 스레드) — 빠른 모드는 코덱 디코딩이 없어 영향이 적습니다.

자주 묻는 질문

원본 영상이 서버로 업로드되나요?
전송되지 않습니다. 영상은 브라우저 안에서만 ffmpeg.wasm(WebAssembly)로 처리되며, 사이트 운영자도 내용을 볼 수 없습니다. 첫 사용 시 ffmpeg 라이브러리(~32MB)만 다운로드되고, 이후 영상 자체는 외부로 나가지 않습니다.
빠른 모드와 정확 모드 차이가 뭔가요?
빠른 모드는 영상 코덱을 그대로 두고(재인코딩 X) 잘라내기 때문에 1GB 영상도 보통 수 초~십몇 초로 끝납니다. 단 H.264의 키프레임(보통 1~5초 간격) 단위로만 정확히 잘리며, 그 사이의 시각을 지정하면 가까운 키프레임으로 끌려갑니다. 정확 모드는 지정한 시각 그대로 프레임 단위로 자르되 H.264로 재인코딩 — 시간이 더 걸립니다.
시각은 어떻게 입력하나요?
HH:MM:SS / MM:SS / 초 — 세 가지 형식 모두 인식합니다.
예: 1:23:45 = 1시간 23분 45초 / 4:30 = 4분 30초 / 90 = 90초. 소수점도 허용 — 90.5는 90.5초.
업로드한 영상의 현재 재생 위치를 시작/끝으로 자동 입력하는 버튼도 사용할 수 있습니다.
출력 포맷은 무엇인가요?
출력은 MP4로 통일됩니다. 카톡·이메일·SNS·QuickTime 어디서든 재생 가능. 빠른 모드는 입력 코덱이 H.264·AAC가 아니어도 컨테이너만 MP4로 바꿔 저장합니다(원본 코덱 그대로 패키징).
처리 시간이 얼마나 걸리나요?
빠른 모드: 영상 크기와 거의 무관 — 1GB 영상도 수 초~십몇 초. 정확 모드: 자르는 구간 길이의 0.3~1배(데스크톱 Chrome). 모바일은 2~3배 더 걸립니다.
큰 영상도 처리 가능한가요?
데스크톱 Chrome 기준 1~2GB까지 처리 가능합니다. iOS Safari·모바일 브라우저는 메모리 한계로 ~500MB 이상에서 불안정할 수 있습니다.
