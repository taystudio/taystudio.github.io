💡 GA4가 JS 주입 OK인 이유

analytics는 client-side만 동작. 검색봇이 GA를 실행할 일도, 실행해서도 안 됨. 따라서 "검색봇이 JS 미실행이라 정적 HTML 필수" 정책이 GA4에는 적용 X. 반대로 verification meta·canonical·OG는 검색봇이 첫 fetch에 봐야 하니 JS 주입 X 정책 유지.

도메인 migration 시작 전부터 통합되어 있던 코드:

// common/site-chrome.js line 137-148
(function () {
  const GA_ID = 'G-79C40NJRYT';
  const s = document.createElement('script');
  s.async = true;
  s.src = 'https://www.googletagmanager.com/gtag/js?id=' + GA_ID;
  document.head.appendChild(s);
  window.dataLayer = window.dataLayer || [];
  window.gtag = function () { window.dataLayer.push(arguments); };
  window.gtag('js', new Date());
  window.gtag('config', GA_ID);
})();

모든 페이지의 head에 <script src="/common/site-chrome.js" defer></script> 로드 → 사용자 페이지 진입 시 IIFE 즉시 실행 → gtag.js 동적 주입 → GA4로 hit 전송.

⚠ Measurement ID는 절대 새로 발급받지 마

URL 변경 시 콘솔이 "새 측정 ID 발급받아야 합니다" 안내 띄우면 그건 새 데이터 스트림 만들기 모드일 때만. 기존 스트림의 URL 수정은 ID 유지됨. 만약 새 ID 발급되면:

  • common/site-chrome.js line 138 ID 갱신 + commit/push 필요
  • 옛 ID로 들어오던 데이터 끊김 → 데이터 연속성 깨짐
  • → cancel 후 "기존 스트림 편집" 경로로 다시 시도

개념 데이터 스트림 URL의 진짜 의미
