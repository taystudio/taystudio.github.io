📚 DNS · A · CNAME — 어떻게 작동하나

migration 따라가기 전에 핵심 개념. dig·nslookup 출력 해석에도 도움.

1. 일상 비유 — 전화번호부

  • 도메인 = 사람 이름 ("김철수")
  • IP 주소 = 그 사람 전화번호 ("010-1234-5678")
  • DNS = 전화번호부 — 이름 → 번호 변환
  • Nameserver = 그 변환표를 보관·응답하는 서버 (우리 = Cloudflare)

사람 이름은 외우기 쉬운데 전화번호는 어렵다. 컴퓨터는 반대 — IP만 알면 통신 가능, 도메인은 못 씀. DNS가 그 사이의 통역자.

3. A record vs CNAME record — 두 종류의 답

A
도메인 → IP 직접 매핑
taystudios.com A 185.199.108.153
taystudios.com A 185.199.109.153
taystudios.com A 185.199.110.153
taystudios.com A 185.199.111.153
한 도메인에 A 4개 = round-robin / anycast 부하 분산. GitHub Pages는 4 IP 모두 같은 서버 풀로 anycast(가장 가까운 데이터센터로 자동 라우팅).
CNAME
도메인 → 다른 도메인 (alias)
www.taystudios.com CNAME taystudio.github.io
Resolver가 이걸 받으면 "그 다른 도메인의 A를 다시 찾아라"고 인식 → taystudio.github.io의 A를 받아 결국 같은 IP 4개 도착. 한 단계 더 거치지만 앞 도메인 IP가 바뀌어도 자동 따라감 (alias의 장점).
⚠ apex(zone root)에는 CNAME 못 씀 — RFC 1912
taystudios.com 자체에는 CNAME 사용 X (다른 record들과 충돌). www·blog 등 서브도메인은 CNAME OK. 그래서 우리 케이스 = apex는 A 4개 / www는 CNAME. (Cloudflare는 "CNAME flattening"으로 apex CNAME 흉내 가능하지만, GitHub Pages는 A 4개를 공식 권장.)

4. 우리 사이트의 실제 흐름 — 두 경로

경로 A — taystudios.com 직접 접근

1 브라우저: "taystudios.com의 A?"
2 Cloudflare NS → "185.199.108.153 (등 4개)"
3 브라우저: 그 IP로 HTTPS 연결 + 헤더 Host: taystudios.com 전송
4 GitHub Pages: Host 헤더 보고 → CNAME 파일에 taystudios.com 등록된 repo 매칭 → studio repo 응답

경로 B — www.taystudios.com 접근 (CNAME 경유)

1 브라우저: "www.taystudios.com의 A?"
2 Cloudflare NS → "CNAME taystudio.github.io야, 그쪽 A 다시 찾아"
3 Resolver: "taystudio.github.io의 A?" → GitHub NS → "185.199.108.153 등"
4 브라우저: 그 IP로 HTTPS + 헤더 Host: www.taystudios.com
5 GitHub Pages: 301 redirect → https://taystudios.com/ (apex로 자동 정규화)

5. Host header — 같은 IP가 어떻게 수백만 사이트를 구별하나

GitHub의 IP 185.199.108.153는 모든 GitHub Pages 사이트가 공유. 그럼 어떻게 어느 repo 응답할지 알까? → HTTP 요청에 붙는 Host 헤더를 보고 분기.

# 브라우저 → IP 185.199.108.153로 보낸 실제 HTTP 요청
GET / HTTP/2
Host: taystudios.com     ← GitHub이 이 헤더 보고 분기
User-Agent: Chrome/...

# GitHub Pages 내부 매칭 (개념상)
if Host == "taystudios.com":
    find repo where CNAME 파일 == "taystudios.com"
    → studio repo (taystudio 계정)
    serve static files
💡 그래서 repo의 CNAME 파일이 결정적
DNS는 "어느 IP로 가라"까지만 안내. 거기서 어느 사이트인지 식별하는 건 Host 헤더 + GitHub 내부 매핑. 그 매핑의 핵심이 repo 루트의 CNAME 파일 — 단순 텍스트 1줄(taystudios.com)이지만 "이 도메인은 이 repo야"의 유일한 선언.

6. TTL · DNS 전파 · 캐시

  • TTL (Time To Live) — DNS 응답에 붙는 "이 답을 N초 동안 캐시해도 됨" 메타. Cloudflare 기본 = Auto (보통 300초)
  • DNS 전파 — 새 record 등록·변경 후 전 세계 resolver 캐시가 비워질 때까지의 시간. 최악 24h, 평균 30분~수시간
  • 전파 빠르게 보고 싶으면 dig @1.1.1.1 taystudios.com 또는 @8.8.8.8로 글로벌 resolver 직접 쿼리. 본인 ISP 캐시는 가장 늦게 갱신될 수 있음
  • 변경 직전 TTL을 60초로 낮추면 전파 가속 가능. 단 Cloudflare Auto면 평소 충분

7. 헷갈리는 3 역할 — Registrar · Nameserver · DNS host

역할하는 일우리 경우
Registrar도메인 소유권 등록·갱신 (ICANN 인증 업체)Cloudflare Registrar
Nameserver (NS)"이 도메인의 DNS 답은 어디서 받아라" 지정 — 도메인 등록 정보의 일부Cloudflare ns (자동)
DNS host실제 A · CNAME · MX 등 record를 보관·응답Cloudflare DNS (자동)
웹서버HTTP 요청 받아 컨텐츠 응답GitHub Pages

Cloudflare Registrar에서 사면 위 3개가 자동 통합. 외부에서 사서 옮긴다면 Registrar에서 NS만 Cloudflare로 변경하면 DNS host도 Cloudflare가 됨. 4번째(웹서버)만 GitHub Pages에 위임.

1 Cloudflare DNS 설정

대시보드 → taystudios.com → DNS · Records → Add record

① A record 4개 + CNAME 1개 추가

GitHub Pages apex domain은 anycast IP 4개로 분산. 모두 같은 이름 @(또는 taystudios.com)으로 추가.

DNS / taystudios.com
+ Add record
Type Name Content Proxy status TTL
A taystudios.com 185.199.108.153 DNS only Auto
A taystudios.com 185.199.109.153 DNS only Auto
A taystudios.com 185.199.110.153 DNS only Auto
A taystudios.com 185.199.111.153 DNS only Auto
CNAME www taystudio.github.io DNS only Auto
⚠ Proxy status — "DNS only"(회색 구름) 권장
주황 구름(Proxied)으로 두면 Cloudflare가 자체 cert 사용 → GitHub Pages Let's Encrypt와 충돌 가능. 또한 SSL/TLS 모드를 Full로 명시적으로 바꿔야 https↔http loop 회피. 캐싱·DDoS 보호가 절실해질 때(트래픽 폭증 후) 켜기. 처음엔 회색이 안전.

② SSL/TLS 모드 확인

좌측 메뉴 SSL/TLS → Overview → 모드 = Full. Flexible은 절대 X (브라우저↔Cloudflare HTTPS, Cloudflare↔GitHub HTTP → mixed content + redirect loop).

③ 등록처(Registrar) 정보 확인

Cloudflare Registrar에서 산 도메인은 자동으로 Cloudflare nameserver(*.ns.cloudflare.com)를 사용 — 별도 NS 변경 불필요. 외부 등록처에서 사서 Cloudflare로 가져온 거라면 Cloudflare가 안내한 NS로 등록처에서 변경 필요.

💡 DNSSEC 활성화 (선택)
DNS → Settings → DNSSEC → Enable DNSSEC. DS 레코드 자동 발급. 등록처 = Cloudflare Registrar면 자동 적용. 보안 시그널이라 나쁠 거 없음.

2 GitHub Pages CNAME 파일 + Settings

두 가지 동시 진행 — repo에 CNAME 파일 1줄 + Settings UI에서 custom domain 입력. UI에서 입력하면 자동으로 파일도 commit됨.

① repo 루트에 CNAME 파일 추가

# studio repo 루트
$ cat CNAME
taystudios.com

딱 1줄. 끝에 newline 1개. 다른 거 X.

② GitHub.com → Settings → Pages

  1. Custom domain 입력란에 taystudios.com 입력 → Save
  2. "DNS check in progress" → "DNS check successful" 까지 대기 (DNS 전파 완료 후 5~30분)
  3. "Enforce HTTPS" 체크박스 활성화되면 체크 (cert 발급 후에만 활성화됨)
⚠ DNS check 실패 시
① Cloudflare에서 4 A records가 정확히 등록됐는지 재확인 ② Proxy = DNS only(회색) 인지 확인 ③ dig taystudios.com +short로 4 IP 모두 응답 오는지 확인 ④ 전파 안 됐으면 1~24h 대기. 보통 30분 이내 처리됨.

③ www 서브도메인 처리

GitHub Pages는 www.taystudios.com → taystudios.com으로 자동 redirect 처리. 별도 설정 불필요. www CNAME만 등록되어 있으면 됨.

3 HTTPS 작동 확인

DNS 전파 + cert 발급 완료 시점. 4 IP 모두 응답 + https://taystudios.com 200 OK 확인.

# 2. www CNAME 확인

# 3. HTTPS 200 OK
$ curl -I https://taystudios.com

# 4. www → apex redirect 확인
$ curl -I https://www.taystudios.com

# 5. http → https redirect 확인

실행 순서

  2. 로컬에서 scripts/build-sitemap.sh 재실행 → 새 도메인 sitemap 생성
  3. IndexNow ping은 일시 보류 — DNS 안정화 후 별도 실행
  4. commit + push → GitHub Pages 배포 완료 (~1분)
  5. 새 도메인에서 sample 페이지 5개 정도 열어 OG 프리뷰 확인
  6. IndexNow 새 도메인으로 67 URL 일괄 ping

taystudio.github.io를 disable하지 않음. GitHub Pages가 자동으로 새 도메인으로 redirect 처리(custom domain 설정 후) → 옛 백링크·북마크·검색결과가 자연 이전. 의도적 deindex는 X.

Cloudflare Proxy (주황 구름) 켤 시점

  • 월 방문 5,000+ 또는 DDoS 위협 감지 시
  • SSL/TLS 모드 = Full 필수 (Flexible 절대 X)
  • Always Use HTTPS · Auto Minify · Brotli 옵션 ON 권장
  • "Page Rules"로 www.* → apex 강제 redirect 추가 가능 (GitHub 자동 외 보강)
  • 주의: Cloudflare가 자체 cert 발급 → GitHub Pages cert와 병존. 옛 cert pinning 클라이언트 거의 없으니 문제 X
