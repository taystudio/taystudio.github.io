차이점 Naver vs Google — 미리 알아두기

같은 검증 콘솔이지만 운영 모델이 꽤 달라. 인덱싱 가속에 직결되는 점들.

항목Google Search ConsoleNaver Search Advisor
인증 후 sitemap 자동 픽업 강함 (robots.txt 자동 감지) 약함 — 수동 제출 필수
인덱싱 속도 자연 크롤 빠름 느림 — "웹페이지 수집 요청" 수동 push 필수
주소 변경(Change of Address) 기능 있음 — 시그널 자동 이전 없음 — 301 redirect로 자연 이전만
일일 색인/수집 요청 한도 충분 도메인당 약 50건/일 (빡빡)
한국 검색 점유율 약 60% 약 30% (필수 채널)
📌 시사점
Naver는 인증 끝나도 끝난 게 아님 — sitemap 제출 + 핵심 URL 수집 요청까지 해야 인덱싱 시작. 한국 트래픽 30%가 걸려 있으니 작업 빠뜨리면 손해 큼.

실행 단계별 진행

사용자 액션 = SA UI 클릭 / 자동 처리 = repo 코드 edit + commit/push.

1
신규 사이트 등록 — HTML 태그 검증
사용자 액션 searchadvisor.naver.com → 웹마스터 도구 → 사이트 등록
  • 로그인 후 우측 상단 "웹마스터 도구" 진입
  • 상단 입력란에 https://taystudios.com → "등록"
  • 검증 방식 두 옵션 중 "HTML 태그" 선택 (head meta 일관성)
  • Naver가 발급한 검증 코드: 
    <meta name="naver-site-verification" content="4cdee18ea9529c885139c0d5971830541da2b23c" />
2
검증 메타 추가 — 옛 코드 유지 병행
자동 처리 index.html 옛 naver 메타 다음 줄에 삽입

옛 사이트(taystudio.github.io)도 SA에 그대로 유지해야 인덱스 자연 이전이 추적됨. 따라서 옛 검증 메타 그대로 두고 신규 메타를 추가. Naver도 동일 name 메타 중복 허용.

<meta name="google-site-verification" content="X6nU3Kz5e0V_..."> <meta name="google-site-verification" content="uiMTiQjh9FfPJu7..."> <meta name="naver-site-verification" content="bfe7d4c2ed7faf..."> <!-- 옛 --> <meta name="naver-site-verification" content="4cdee18ea9529c..."> <!-- 신규 --> <meta name="msvalidate.01" content="064396E810073C47308AEC058EDC3846" />
3
commit + push — GitHub Pages 배포
자동 처리 commit c170f15 
git add index.html
git commit -m "feat: Naver SA 신규 도메인 property 검증 meta 추가"
git push origin main

배포 polling — 약 30~40초 후 라이브 반영:

$ curl -sS https://taystudios.com/ | grep naver-site-verification
  <meta name="naver-site-verification" content="bfe7d4c2ed7faf...">   ← 옛
  <meta name="naver-site-verification" content="4cdee18ea9529c..."> ← 신규
4
"소유 확인" 클릭 → 검증 통과
사용자 액션 발급 화면에서 "소유 확인" 버튼

Naver가 https://taystudios.com/ fetch → head 안에서 4cdee18ea9529c... 검증 코드 발견 → 즉시 통과. GSC와 달리 주소 변경 자동 유도 화면은 없음. 검증 통과 후 바로 사이트 대시보드로 진입.

다음 인덱싱 가속 작업 (필수)

Naver는 인증만으로 끝나지 않음. 다음 4가지 수동 작업이 인덱싱 속도를 결정.

  • 사이트맵 제출 — 좌측 메뉴 요청 → 사이트맵 제출 → 입력란에 sitemap.xml (path만) → "확인"
  • RSS 제출 — 현재 RSS 안 만들었으므로 skip
  • 웹페이지 수집 요청 — 요청 → 웹페이지 수집 요청 → 핵심 URL 수동 push (도메인 50건/일 한도)
  • robots.txt 검증 — 검증 → robots.txt → Sitemap 줄 인지 확인
  • 사이트 최적화 점수 확인 — 검증 → 사이트 최적화 → 80+ 목표 (현재 head 시그널 박혀 있어 만점 가까이 예상)
  • 옛 사이트 정리 — 6개월 후 옛 사이트(taystudio.github.io) 등록 삭제 (지금은 유지)

결정 의사결정 메모

옛 Naver 사이트 유지 — 6개월 정책

Naver는 주소 변경 기능이 없어서 옛 사이트의 인덱스 시그널이 신규로 자동 이전되지 않음. 그래서 옛 사이트 등록을 유지하면서 301 redirect로 Naver 봇이 점진 갱신하도록 둠. 옛 검증 메타도 동일 기간 유지. 6개월 후 옛 사이트 노출/클릭 트렌드가 거의 0이면 옛 사이트 삭제 + 옛 메타 줄 제거.

웹페이지 수집 요청 — 13개 우선순위

50건/일 한도가 있어 전체 sitemap(67 URL) 한 번에 제출 X. 트래픽/매출 기여도 + 카테고리 hub 위주로 13개 추림 (메인 1 + 카테고리 5 + 핵심 도구 7). 나머지 long-tail은 sitemap + 자연 크롤로 위임. 효과 검증은 SA 노출/클릭 통계로 1~2주 후.

옛 사이트 즉시 삭제 금지

옛 사이트 삭제 시 Naver 봇이 redirect 추적 신호를 잃을 수 있음. 자연 이전이 끝났다고 판단되는 시점(6개월 후)에 삭제. 삭제 전에 SA 옛 사이트 분석 화면에서 노출 트렌드 확인 필수.

재현 검증 명령

6개월 후 옛 사이트 정리 시점에 다시 실행해서 상태 확인용.

# 1. 두 검증 메타 라이브 존재 확인
curl -sS https://taystudios.com/ | grep naver-site-verification

# 기대: 두 줄 (옛 + 신규)
#   <meta name="naver-site-verification" content="bfe7d4c2ed7faf...">
#   <meta name="naver-site-verification" content="4cdee18ea9529c...">

# 2. 301 redirect 동작 (Naver 봇이 옛 → 신규 매핑하는 경로)
curl -sSI https://taystudio.github.io/ | head -3
# 기대: HTTP/2 301 Moved Permanently / Location: https://taystudios.com/

# 3. sitemap.xml 정상 응답 (Naver SA 사이트맵 제출 대상)
curl -sSI https://taystudios.com/sitemap.xml | head -3
curl -sS  https://taystudios.com/sitemap.xml | grep -c '<loc>'
# 기대: HTTP/2 200 / 67개 URL
