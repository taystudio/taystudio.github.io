실행 단계별 진행

사용자 입력 = GSC UI 클릭 / 자동 처리 = repo 코드 edit + commit/push.

1
신규 property 추가 — URL prefix 방식
사용자 액션 search.google.com/search-console → "속성 추가"
  • 속성 유형 두 옵션 중 "URL prefix" 선택 (Domain property가 권장이지만 DNS TXT 추가 비용보다 HTML meta 한 줄 추가가 더 가볍다 판단)
  • 입력: https://taystudios.com/
  • 검증 방식: "HTML 태그"
  • Google이 발급한 검증 코드: 
    <meta name="google-site-verification" content="uiMTiQjh9FfPJu7Mx4FxF-hiRG5eq6Pi7qrIYqzR3Nw" />
💡 왜 URL prefix를 골랐나
Domain property는 DNS TXT 레코드 1건 추가 + 전파 1~10분 대기가 필요. URL prefix는 head meta 1줄 추가로 끝. 신규 도메인은 https://taystudios.com/ 하나만 다루면 되므로 (Enforce HTTPS로 http/, www. 자동 통합) URL prefix로 충분.
2
검증 메타 추가 — 옛 코드 유지하며 병행
자동 처리 index.html line 10 다음 줄 삽입

옛 property(taystudio.github.io)는 6개월간 살려둬야 redirect 시그널이 자연 이전됨. 따라서 옛 검증 메타는 그대로 두고 신규 메타를 추가. Google은 head 내 동일 name 메타 중복 허용.

<meta name="theme-color" content="#2563eb"> <meta name="color-scheme" content="light dark"> <meta name="google-site-verification" content="X6nU3Kz5e0V_ZfAHt0QcXhuvw99O9RUCvkVe97B603c"> <!-- 옛 --> <meta name="google-site-verification" content="uiMTiQjh9FfPJu7Mx4FxF-hiRG5eq6Pi7qrIYqzR3Nw"> <!-- 신규 --> <meta name="naver-site-verification" content="bfe7d4c2ed7faf9c9204f614640698ae5bc7b4cd">

4
GSC "확인" 클릭 → 신규 property 검증 통과
사용자 액션 발급 화면에서 "확인" 버튼

Google이 https://taystudios.com/ fetch → head 안에서 uiMTiQjh9FfPJu7... 검증 코드 발견 → 즉시 통과.

검증 통과 후 GSC가 자동으로 옛 property(taystudio.github.io)에서의 주소 변경(Change of Address) 화면으로 유도. 옛 property에 인덱스 시그널이 많이 쌓여 있을 때 Google이 추천하는 경로.

5
주소 변경(Change of Address) 유효성 검사
Google 자동 검증 옛 property → 신규 property 매핑 자격 확인

Google이 자동으로 다음 4가지 검사를 수행. 결과 모두 통과:

✅ 현재 상태 — "이전 중" (Google 자동 처리 중)
Google이 옛 property의 인덱스 시그널을 신규 property로 점진 이전. 검색 결과의 URL도 옛 → 신규로 자연 전환. 처리 기간 = 수일~수주(주요 URL) / 6개월(전체 long-tail).

다음 미완료 액션

신규 property 검증과 주소 변경은 끝났지만, 신규 도메인의 인덱싱 가속을 위해 추가 작업 필요.

  • 신규 property 좌측 메뉴 Sitemaps → "sitemap.xml" 입력 → 제출 (path만, 전체 URL 아님)
  • 신규 property 좌측 메뉴 URL 검사 → https://taystudios.com/ 색인 생성 요청 (메인부터)
  • 동일하게 5 카테고리 hub URL 색인 요청 (/tools/·/text/·/image/·/pdf/·/video/)
  • 주요 도구 페이지 5~10개 추가 색인 요청 (일일 한도 내에서)
  • 6개월 후 — 옛 property에서 인덱싱 트래픽이 거의 0에 수렴하는지 확인 후 옛 property 정리(삭제)

결정 의사결정 메모

URL prefix vs Domain property — URL prefix 채택

이유

  • Enforce HTTPS + www → apex 301로 origin이 https://taystudios.com/ 단 하나로 수렴
  • HTML meta 추가 비용 1줄 < DNS TXT 등록 + 전파 대기
  • 이미 head에 동일 패턴(google-site-verification, naver-site-verification, msvalidate.01) 박혀 있어 일관성
옛 검증 메타 유지 — 6개월 정책

옛 property(taystudio.github.io)는 인덱스 시그널 자연 이전을 위해 최소 6개월 유지. 따라서 옛 검증 메타도 동기간 유지.
이전 진행 모니터링 = GSC 옛 property에서 노출수/클릭수 트렌드 확인. 거의 0에 수렴하면 옛 property 삭제 + 옛 메타 줄도 제거.

옛 property 즉시 삭제 금지

신규 property 검증 통과했다고 옛 property를 바로 지우면 주소 변경 시그널이 끊김. Google은 옛 property에서 redirect를 계속 관찰해야 신규로의 권위 이전을 확신 처리. 6개월 = 일반적 권장 기간.

재현 검증 명령

6개월 후 메타 회수 시점에 다시 실행해서 상태 확인용.

# 1. 두 검증 메타 라이브 존재 확인
curl -sS https://taystudios.com/ | grep google-site-verification

# 기대: 두 줄 (옛 + 신규)
#   <meta name="google-site-verification" content="X6nU3Kz5e0V_...">
#   <meta name="google-site-verification" content="uiMTiQjh9FfPJu7...">

# 2. 301 redirect 동작
curl -sSI https://taystudio.github.io/ | head -3
# 기대: HTTP/2 301 Moved Permanently / Location: https://taystudios.com/

# 3. Enforce HTTPS
curl -sSI http://taystudios.com/ | head -3
# 기대: HTTP/1.1 301 Moved Permanently / Location: https://taystudios.com/
