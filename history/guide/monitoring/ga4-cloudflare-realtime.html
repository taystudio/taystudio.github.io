2. Cloudflare Web Analytics — 5분 셋업

도메인이 이미 Cloudflare에 위임돼 있으면 script 1줄만 추가하면 끝. 무료 · 분 단위 · 쿠키리스.

2-A. 셋업 절차

1 Cloudflare 대시보드 진입

https://dash.cloudflare.com 로그인 → 좌측 상단 계정 선택.

좌측 사이드바 Analytics & Logs 펼치기 → Web Analytics.

3 + Add a site 클릭 → 도메인 입력 → RUM 모드 선택

호스트네임: taystudios.com 입력. 다음 화면에서 RUM 옵션 4가지 중 선택:

  • Enable — Cloudflare edge에서 자동 주입. Proxied(주황 구름) 도메인 전용. token 안 보여줌
  • Enable, excluding visitor data in the EU — 위와 동일하지만 EU 사용자 제외
  • Enable with JS Snippet installation — 수동 삽입. token + script tag 발급 ★ 우리가 선택
  • Disable — 끔
왜 Manual인가 GitHub Pages는 자체 CDN(Fastly) + Let's Encrypt 자동 갱신을 쓰는데, Cloudflare proxy(주황 구름)를 켜면 SSL 종료가 Cloudflare에서 일어나 ACME challenge 통과 실패 → 인증서 갱신 깨질 수 있음. 그래서 GitHub 권장 = DNS only(회색 구름). proxy off면 Cloudflare가 트래픽을 못 보니 자동 주입 X → Manual 필수.
4 Beacon token 발급 → 스크립트 복사

Cloudflare가 다음 형태의 script tag를 줌:

<script defer src="https://static.cloudflareinsights.com/beacon.min.js"
        data-cf-beacon='{"token": "abc123def456..."}'></script>

이 token이 사이트별 식별자.

5 site-chrome.js의 GA4 블록 옆에 추가

Cloudflare가 준 원본 tag:

<script defer src='https://static.cloudflareinsights.com/beacon.min.js'
        data-cf-beacon='{"token": "412699582bf74c0bbb767378e033c0ce"}'></script>

모든 페이지에 정적으로 삽입하지 않고 common/site-chrome.js의 GA4 IIFE(:185) 직후에 같은 동적 로드 패턴으로 추가 — 1곳 수정으로 전 페이지 적용:

(function () {
  const s = document.createElement('script');
  s.defer = true;
  s.src = 'https://static.cloudflareinsights.com/beacon.min.js';
  s.setAttribute('data-cf-beacon', '{"token": "412699582bf74c0bbb767378e033c0ce"}');
  document.head.appendChild(s);
})();
왜 setAttribute인가 s.dataset.cfBeacon = ...도 동일하게 동작하지만 JSON 문자열을 그대로 보존하기 위해 setAttribute를 사용. 빈 dataset 키 변환(camel→kebab) 헷갈림 방지 목적도 있음.

token은 client-side에 노출되지만 beacon 특성상 정상(모든 방문자 브라우저가 받아감 — 비밀이 아님).

6 commit · push → 5분 후 데이터 수집 시작

GitHub Pages 배포 후 본인 또는 다른 디바이스로 사이트 방문 → 5~10분 후 Cloudflare 대시보드에 페이지뷰 카운터 1↑ 확인.

3. 두 도구 어떻게 같이 쓰나

GA4 — 깊이

  • 이벤트 추적 (page_view·click·scroll·custom)
  • 전환 funnel·attribution
  • 사용자 행동 시퀀스 (탐색)
  • 디바이스 ID 기반 cohort
  • Realtime 30분 윈도우

Cloudflare — 즉시성

  • 분 단위 업데이트
  • 광고차단 무관 (잡힘)
  • 쿠키 0 → GDPR 동의 불요
  • 심플한 referrer / pages / devices
  • 가벼움 (3KB 미만)

3-B. 데이터 차이가 나면?

  • Cloudflare 200, GA4 130 → 정상. 광고/쿠키차단 사용자 미수집 차이 (보통 GA4가 70~80%)
  • 두 곳 모두 갑자기 ↓ → 실제 트래픽 감소 또는 sitemap·검색엔진 색인 문제. Search Console 점검
  • GA4만 갑자기 ↓ → GA4 셋업 깨짐. site-chrome.js GA_ID·gtag 호출 점검, GA 콘솔 "데이터 스트림" 상태 확인
  • Cloudflare만 갑자기 ↓ → beacon script 에러. DevTools Network 탭에서 cloudflareinsights.com request 확인

5. Troubleshooting

GA4 Realtime에 활성 사용자 0인데 사이트 트래픽 있음

  • 본인 브라우저에서 사이트 열어 자가 테스트 → 1분 내 Realtime에 1↑ 떠야 함
  • 안 뜨면 GA가 측정 안 되는 것. DevTools Console에 gtag is not defined 같은 에러 확인
  • common/site-chrome.js:185-193 GA4 init 블록 정상 실행되는지 점검 — window.dataLayer·window.gtag 존재해야 함
  • 광고차단 ON이면 본인이 안 잡힐 수 있음. 시크릿 창 + 차단 OFF로 재시도

Cloudflare Web Analytics에 0이 계속 뜸

  • script가 GitHub Pages에 배포됐는지 — view-source:로 HTML 확인. cloudflareinsights.com beacon 보여야 함
  • token이 도메인과 매칭 — Cloudflare 대시보드 site detail에서 token 재확인
  • 도메인 hostname mismatch — www.taystudios.com vs taystudios.com 등록 차이 확인
  • 5분 캐시 — push 직후 즉시 안 뜸. 10분 후 재확인

(direct) / (none) 비율이 너무 높다

  • 북마크·URL 직접 입력 = 정상
  • 인앱 브라우저(KAKAO·Line·NAVER inapp)에서 referrer 누락 = 흔함
  • HTTPS → HTTP 점프(이젠 거의 없음) = referrer drop
  • 50% 넘으면 의심 — UTM 파라미터 사용 또는 referrer-policy meta 점검
