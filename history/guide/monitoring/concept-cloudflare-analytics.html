Cloudflare Web Analytics 개념정리 — 대시보드 읽는 법

2026-05-10 작성 / 카테고리: 운영 가이드 / 짝 문서: GA4 + Cloudflare 실시간 모니터링 가이드

한 줄 정리

  • RUM = Real User Measurement. 사용자 브라우저에 박힌 beacon이 보내준 실제 측정값(서버 로그 X)
  • 핵심 5지표 — Visits / Page views / Load time / Core Web Vitals / Top referrers
  • 샘플 크기 100+ 쌓인 후 분포 신뢰. 30 미만은 거의 노이즈
  • GA4 대비 Cloudflare 수치가 보통 10~30% 더 큼 — 광고차단·쿠키차단 사용자도 잡혀서
  • 본인 트래픽 자동 제외 X — IP 필터 기능 없음. 셀프 테스트 PV는 알아서 빼고 보기

1. RUM이 뭔가 — 작동 원리

전통적 분석 도구는 두 종류:

방식측정 위치장점단점
서버 로그웹서버차단 불가, 봇 포함 모두 잡힘brower-side(클릭·LCP) 못 봄
RUM (client-side)방문자 브라우저실제 사용자 경험·LCP·CLS 측정 가능광고차단·쿠키차단 시 누락

Cloudflare Web Analytics = RUM. beacon.min.js가 페이지 로드 후 사용자 브라우저에서 자동 실행되어 다음 정보를 cdn-cgi/rum로 POST:

  • 현재 URL, referrer, viewport·디바이스·브라우저
  • 페이지 로드 시간 (Performance API 기반)
  • Core Web Vitals (LCP, INP, CLS)
  • 이벤트 (페이지뷰 1건)

쿠키 0 → GDPR 동의 불요. 하지만 광고차단 확장(uBlock·AdGuard 등)이 cloudflareinsights.com 도메인을 차단하면 측정 안 됨.

2. 핵심 지표 5개 — 정확히 무엇을 세는가

Visits 방문

한 사용자의 한 세션. 30분 비활성 시 다음 PV는 새 Visit으로 카운트.

  • 방문자가 10페이지 연속 본 경우 = Visit 1, PV 10
  • 방문자가 30분 자리 비웠다 다시 본 경우 = Visit 2
  • "오늘 우리 사이트에 몇 명이나 왔나" 추정치 (정확한 unique user는 쿠키리스라 불가)

Page views 페이지뷰

페이지 로드 발생 횟수. 같은 사용자가 새로고침하면 +1, 다른 페이지로 이동하면 +1.

  • 방문자가 홈 → 계산기 → 텍스트 → 홈 = PV 4
  • PV / Visits = 평균 페이지/방문. 우리 사이트는 보통 2~4가 건강한 범위 (utility 사이트 특성)
  • 1.0에 가까우면 = 첫 페이지에서 다 이탈 (bounce). SEO 문제 또는 콘텐츠 매치 안 맞음 신호

Page load time 평균 로드 시간

Performance API의 navigationStart부터 loadEventEnd까지. 즉 페이지가 "다 떴다" 시점까지의 ms.

  • ≤ 500ms = 매우 빠름 (정적 사이트 + CDN 효과)
  • 500~1500ms = 정상
  • 1500~3000ms = 느림. AdSense 등 외부 script 영향 의심
  • > 3000ms = 큰 문제. 이미지·script·폰트 점검
주의 load time은 사용자 네트워크·기기 영향을 그대로 받음. 4G 모바일 사용자가 많아지면 자연히 ↑. "내 환경에서 빠른데 왜 평균이 느리지?" = 다른 사용자 환경 때문.

Core Web Vitals LCP · INP · CLS

Google이 정의한 사용자 경험 3대 지표. SEO 랭킹에 직접 영향 (Google Search 신호 중 하나). §3에서 자세히.

Top referrers 상위 유입 출처

방문자가 어디서 우리 사이트로 왔나. ★ TAYSTUDIO에서 가장 중요한 패널.

  • google.com, www.google.com = 구글 검색결과
  • search.naver.com, m.search.naver.com = 네이버 검색 (PC/모바일)
  • bing.com = 빙 검색
  • (direct) = referrer 없음 (북마크·URL 직접·앱 in-app browser·메신저 링크)
  • t.co = 트위터/X, l.facebook.com = 페이스북, l.instagram.com = 인스타

각 referrer의 비율 변화가 핵심 신호 — "구글 비중 60% → 40% 떨어짐 + 네이버 5% → 25% 상승" 같은 변화는 검색엔진별 색인 진척도 추정 가능.

3. Core Web Vitals — 임계값과 해석

Google이 정한 "좋은 사용자 경험" 3대 지표. 각 페이지뷰마다 측정 → 분포로 표시. "LCP Good 91%"는 PV의 91%가 LCP "Good" 임계값을 통과했다는 뜻.

3-A. LCP — Largest Contentful Paint

화면에서 가장 큰 콘텐츠(보통 hero 이미지 또는 큰 텍스트 블록)가 렌더된 시점. "눈에 보이는 페이지가 떴다"의 객관적 측정.

Good ≤ 2.5초
Needs Improvement ≤ 4.0초
Poor > 4.0초
  • 좋게 만들려면: hero 이미지 작게, lazy 로딩 안 걸기, font-display: swap, AdSense 지연 로드
  • Poor 비율 10% 이하면 합격선. 우리 사이트는 91% Good = 매우 좋음

3-B. INP — Interaction to Next Paint

사용자가 클릭·탭·키 입력했을 때 화면이 응답하기까지의 시간. 2024년부터 FID 대신 사용되는 새 지표.

Good ≤ 200ms
Needs Improvement ≤ 500ms
Poor > 500ms
  • JavaScript 실행이 메인 스레드를 막으면 ↑. 무거운 라이브러리·동기 처리·이미지 변환(WASM 포함) 시 주의
  • 우리 사이트 video·image upscale 같은 client-side 처리 페이지는 처리 중 클릭 응답 ↓ 가능

3-C. CLS — Cumulative Layout Shift

페이지 로드 중 레이아웃이 갑자기 점프하는 양. 광고가 늦게 떠서 본문 밀려나거나, 이미지 width/height 안 박혀있어 스페이스가 갑자기 생기는 경우 발생. 무차원 점수.

Good ≤ 0.1
Needs Improvement ≤ 0.25
Poor > 0.25
  • 좋게 만들려면: 이미지·iframe에 width/height 또는 aspect-ratio 명시, AdSense 슬롯 영역 미리 확보, 동적 콘텐츠는 위에서 아래로만 추가
  • AdSense 승인 후 광고가 본문 위에 끼면 CLS 급등 위험 — 슬롯 사이즈 고정 필수

3-D. "Poor 9%"가 의미하는 것

스크린샷에서 본 LCP "Poor 9%" — 이건 "측정된 페이지뷰의 9%가 LCP가 4초를 넘었다"는 뜻. 11 PV 중 1건 = 9.09%. 통계적으로 의미 없는 작은 N.

샘플 크기 효과 11 PV에서 "Poor 9%"는 그냥 "1건이 느렸다"일 뿐. 100 PV·1000 PV 쌓여도 9% 유지되면 그때부터 진짜 문제. 초기에는 절대 수치보다 추세를 봐야 함.

4. Top referrers 깊게 읽기

4-A. (direct)가 너무 많아 보여도 정상일 수 있다

다음 경로는 모두 (direct)로 잡힘:

  • 북마크 클릭, URL 직접 입력
  • 카카오톡·라인·텔레그램 링크 (대부분 referrer 안 보냄)
  • iOS Safari ITP로 인한 referrer 누락
  • HTTPS → HTTPS 점프 시 일부 referrer-policy 설정에서 누락
  • 인앱 브라우저 (NAVER·KAKAO·Instagram 인앱)
  • Google AMP 캐시 거쳐서 들어온 경우

(direct) 비율 30~50% = 정상. 60% 넘으면 대체로 모바일·앱 트래픽 비중 ↑. 80% 넘으면 referrer-policy meta 점검 권장.

4-C. 검색 키워드는 안 보임

"네이버에서 어떤 단어로 들어왔나"는 Cloudflare Web Analytics에서 못 봄. 검색엔진이 referrer URL에서 query 파라미터를 의도적으로 빼서 보냄(privacy 표준). 키워드 단위 분석은 각 검색엔진의 webmaster console에서만 가능:

  • Google → Search Console
  • Bing → Webmaster Tools
  • Naver → Search Advisor

5. 통계적 유의성 — N에 따라 다르게 읽기

샘플 크기 (PV)신뢰도봐야 할 것
0~30거의 노이즈"동작하나"만 확인. 비율 신뢰 X
30~100방향성 추정top 1~2개 referrer만 의미. 작은 비중은 무시
100~500약한 신뢰분포 대략 OK. 5% 미만은 noise 가능
500~5000실용적 신뢰모든 카테고리 비교 의미. 추세 확인 가능
5000+강한 신뢰day-of-week·시간대까지 의미 있음

지금 단계(시작 직후, PV ~10) = "동작하나" 확인 단계. 비율·percentage는 무시하고 "데이터가 흘러들어오나"만 보면 됨.

6. GA4와 차이가 나는 이유

같은 사이트인데 두 도구 수치가 다름. 정상.

차이 패턴원인
Cloudflare 100, GA4 70~80광고차단·쿠키차단 사용자 — Cloudflare는 잡고 GA는 못 잡음. 정상
두 곳 모두 유사하게 ↓실제 트래픽 감소 또는 검색엔진 색인 문제
GA4만 갑자기 ↓GA 셋업 깨짐 (gtag 호출 실패, ID 잘못)
Cloudflare만 갑자기 ↓beacon script 차단·token mismatch·도메인 hostname 문제

안전 가정 = 실제 사용자 ≈ Cloudflare 수치, GA4는 그 중 70~80%만 보고 있다고 보면 됨. AdSense 수익 추정 시 GA보다 Cloudflare가 더 가까운 분모.

8. 한계 — Cloudflare가 못 하는 것

  • 본인 트래픽 제외 X — IP 필터 기능 없음. 셀프 테스트 PV가 통계에 섞여 들어감. 데이터 적을 때는 본인 트래픽이 큰 비중 차지하니 감안
  • 이벤트·전환 추적 X — 버튼 클릭·다운로드·폼 제출 카운트 못 함. 이건 GA4가 잘함
  • 사용자 단위 cohort X — 쿠키리스라 같은 방문자 재방문 추적 불가. unique user는 추정치
  • funnel 분석 X — 페이지 A → B → C 흐름 못 봄
  • 키워드 X — referrer만 보임, query 안 보임 (Search Console로)
  • 광고차단 사용자 누락 — 다만 RUM 도구는 모두 같은 한계. GA4보다는 덜 누락

9. 우리 사이트 현재 상태 진단 — 2026-05-10 기준

첫 캡처 (Visits 3, PV 11) 해석:

  • beacon 정상 동작 ✅ — 데이터 흐름 확인됨
  • Page load 340ms ✅ — 매우 빠른 수준. 정적 + GitHub Pages CDN 효과
  • LCP Good 91% ✅ — 1건의 Poor는 통계적으로 무시
  • PV/Visits = 11/3 ≈ 3.7 ✅ — 한 방문당 3.7페이지. 좋은 engagement (utility 사이트 평균 이상)
  • 이 단계 = 본인 자가 테스트 트래픽 — 실제 외부 유입은 검색엔진 색인 진행되며 1~4주 후부터 잡힘

지금은 "셋업 됐나" 확인 끝. 실질적 referrer 분포는 PV 100+ 쌓인 후 다시 봐야 의미 있음.
