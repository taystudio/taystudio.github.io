IndexNow 개념정리

2026-05-07 작성 / 카테고리: SEO 인프라 / 적용: scripts/indexnow-ping.sh

한 줄 정리

  • 본질: HTTP 한 번(POST + GET 검증)으로 검색엔진 인덱스 큐에 push
  • 참여 엔진: Bing · Yandex · Naver(2024 합류) · Seznam · DuckDuckGo. Google 미지원
  • 인프라: 정적 사이트 친화 — key 파일 1개 + 스크립트 1개, 백엔드 0
  • 한계: 인덱스 자체는 보장 X, 가속 채널일 뿐

1. 전체 흐름

운영자 → IndexNow Hub → 참여 검색엔진까지의 시퀀스. 한 번의 POST로 여러 엔진에 동시 전파됨.

[운영자 로컬]              [GitHub Pages]              [IndexNow Hub]              [참여 검색엔진]
                                                       (api.indexnow.org)         (Bing/Yandex/Naver/Seznam)
     │                          │                            │                           │
1.   │ commit + push            │                            │                           │
     ├─────────────────────────>│                            │                           │
     │                          │ {key}.txt 배포                                         │
     │                          │ /sitemap.xml 갱신                                      │
     │                          │                            │                           │
2.   │ POST {host,key,keyLocation,urlList[]}                 │                           │
     ├──────────────────────────────────────────────────────>│                           │
     │                          │                            │                           │
3.   │                          │  GET {keyLocation}         │                           │
     │                          │<───────────────────────────┤  (소유권 검증)             │
     │                          │ 200 OK                     │                           │
     │                          ├───────────────────────────>│                           │
     │                          │  body == key?              │                           │
     │                          │                            │                           │
4.   │ HTTP 200/202 (queued)    │                            │                           │
     │<──────────────────────────────────────────────────────┤                           │
     │                          │                            │                           │
5.   │                          │                            ├──────────────────────────>│
     │                          │                            │  URL 목록 relay            │
     │                          │                            │                           │
6.   │                          │  검색엔진 크롤러 fetch (분~시간)                         │
     │                          │<──────────────────────────────────────────────────────  │
     │                          │                                                         │
7.   │                          │                                       인덱스 큐 진입    │
     │                          │                                  (며칠 → 빠르게 단축)   │

2. 핵심 메커니즘 5가지

① Push 모델

기존 sitemap = "여기에 sitemap 있으니 와서 봐" → 검색엔진이 폴링 주기에 맞춰 fetch (며칠 간격). IndexNow = 변경 즉시 검색엔진에 통보 → 봇이 바로 와서 크롤. 인덱스 시점이 며칠 → 몇 시간으로 단축.

② Key 파일 = 소유권 증명

HTTPS 도메인 검증과 똑같은 로직. https://taystudio.github.io/{key}.txt 가 200 + body에 정확히 그 key 문자열을 가져야 IndexNow가 "이 사람이 이 도메인 진짜 주인"이라고 판정.

외부 공격자가 우리 도메인으로 ping 보내려면 그 도메인에 key 파일 올릴 수 있어야 하는데 = 이미 도메인 장악한 상태라 IndexNow 우회할 필요도 없음. 그래서 key는 공개돼도 무관.

③ Hub & Spoke 구조

원래 Bing·Yandex가 각자 만들었던 ping endpoint를 통합. 지금은:

api.indexnow.org (또는 bing.com/indexnow, yandex.com/indexnow)
  → 한 곳에 ping 보내면
  → Bing이 internal로 다른 참여 엔진(Yandex·Naver·Seznam)에 relay

그래서 Naver Search Advisor 따로, Bing Webmaster 따로 ping 안 보내도 됨. 하나로 끝.

④ HTTP 응답 코드 의미

코드뜻대응
200 OK받았고 검증·queue 통과—
202 Accepted받았음 (검증은 비동기 진행)—
400payload JSON 형식 오류스크립트 점검
403keyLocation에서 key 검증 실패 (파일 없음·내용 불일치)commit+push 확인, curl {keyLocation} 200 확인
422urlList의 URL이 host와 다름sitemap host 점검
429rate limit (이상 빈도로 ping 시)잠시 후 재시도

⑤ 큐잉 → 크롤 → 인덱스 (보장 X)

중요한 한계 ping = "와봐라 신호". 크롤러가 와도 페이지 품질 검증 후 인덱스 결정 = 검색엔진 재량. ping이 인덱스를 보장하지 않음. 인덱스 속도를 끌어올리는 가속 채널일 뿐.

거꾸로 말하면 = 콘텐츠가 좋으면 인덱스 빨라지고, 광고만 잔뜩이거나 thin content면 ping 보내봤자 무시당함. 우리 사이트는 도구별 FAQ + 출처 + 본문 풍부해서 유리.

3. 우리 케이스의 구체 예

bash scripts/indexnow-ping.sh 실행하면 다음 요청이 발송됨:

POST https://api.indexnow.org/indexnow
Content-Type: application/json

{
  "host": "taystudio.github.io",
  "key": "b762f70e61da4ac199b51566e31748b3",
  "keyLocation": "https://taystudio.github.io/b762f70e61da4ac199b51566e31748b3.txt",
  "urlList": [
    "https://taystudio.github.io/",
    "https://taystudio.github.io/tools/",
    "https://taystudio.github.io/tools/storage/",
    ... (총 67개)
  ]
}

→ Bing 서버가 keyLocation GET 해서 body == key 확인 → 67 URL을 Bing·Yandex·Naver·Seznam 큐에 분배 → 각 엔진 크롤러가 알아서 fetch.

4. Naver IndexNow 특이점

네이버는 2024년 합류라 상대적으로 신규. IndexNow ping 받아도 자체 정책으로 추가 검증 후 크롤하는 패턴 관찰됨 (Bing보다 ping → 크롤 lead time 김). 그래도 네이버 Search Advisor sitemap 제출만 하는 것보다 훨씬 빠름 = 한국 시장 ROI 큰 이유.

5. Google이 빠진 이유

Google은 자기 인덱스 인프라(Googlebot + Search Console)에 self-confident해서 IndexNow에 참여 안 함. 대신 Indexing API 따로 있는데, 채용공고·라이브스트림에만 허용 = 일반 utility 사이트는 못 씀.

Google은 sitemap·SC·콘텐츠 품질로만 cover.

6. 보안 — key 공개 무관성

key가 공개돼도 안전한 설계 이유:

  • 공격자가 우리 도메인으로 ping 보내봤자, IndexNow는 우리 GitHub Pages 서버에 keyLocation GET 하는데 우리 서버는 key 파일을 정상 반환하니 검증 통과 → 결과는 "우리 사이트 페이지를 인덱스해라"는 정상 신호. 공격자에게 이득 X.
  • 공격자가 다른 도메인으로 ping 보내면 → 그 도메인의 keyLocation에서 우리 key가 안 나오니 검증 실패.
  • 그래서 공개돼도 안전한 토큰 설계 — robots.txt에도 key location 주석 노출 가능.

8. 적용 단계별 작업 + 의미 (2026-05-07 회고)

이번 적용 작업을 단계별로 분해 — 각 단계가 왜 필요했는지 + 어떤 결정을 내렸는지. 다음에 다른 사이트에 적용하거나 비슷한 push-style 프로토콜(Naver Search Advisor API·Bing Submit URL 등)에 응용할 때 참고용.

① 사전 조사

  • 한 일: scripts/build-sitemap.sh 구조 파악 (CATEGORIES 배열·emit_url·noindex 자동 제외 로직), 현재 sitemap URL 수(67) 확인, robots.txt·repo 루트 구조 확인
  • 왜: 기존 빌드 파이프라인에 ping을 hook할 위치 + URL 소스 결정용. 새 인프라 만들지 말고 기존 sitemap 빌드 흐름에 얹는다가 정적 사이트 정신에 맞음 (단일 진입점)

② Key 발급 + 검증 파일 배치

  • 한 일: uuidgen → 32자 lowercase hex(b762f70e...) 생성. 루트에 {key}.txt 작성, body에 key 1줄
  • 왜: IndexNow 스펙은 8~128자 영숫자 허용. UUID에서 하이픈만 제거하면 32자라 적정. 충돌 가능성 사실상 0(UUID v4 = 122-bit 엔트로피)
  • 왜 루트 직속: keyLocation을 명시적으로 보내긴 하지만, 루트 직속이 IndexNow 관행. 일부 엔진은 https://host/{key}.txt 직접 추측 검증도 함

③ Ping 스크립트 작성 (scripts/indexnow-ping.sh)

  • 한 일: bash 스크립트 99줄. sitemap.xml에서 <loc> 추출 → JSON payload 빌드 → curl POST → HTTP 코드별 분기
  • 왜 standalone: 빌드 스크립트와 분리(단일 책임 원칙). sitemap 빌드 없이 ping만 재시도 가능 (예: 첫 ping 실패 시 디버깅)
  • 왜 --dry-run 옵션: ping은 비가역 작업(전송 후 취소 X). 첫 실행 전 payload 검증 필수. 이후 디버깅 시도 dry-run으로 빠르게 확인 가능
  • 왜 한국어 에러 메시지: 1인 운영이라 메시지 톤 일관 + 에러 코드별 대응 가이드 즉시 노출 → 검색 시간 절약

④ 빌드 스크립트에 hook (build-sitemap.sh +7줄)

  • 한 일: sitemap 생성 후 indexnow-ping.sh 자동 호출. INDEXNOW_PING=0 env로 opt-out
  • 왜 default ON: 새 도구 출시 워크플로우 = "build-sitemap.sh 한 번 실행"으로 끝나야 함. opt-in이면 사용자가 ping 까먹을 위험 → 색인 지연
  • 왜 opt-out 옵션: commit 전 로컬 dry-build·테스트 시 의도치 않은 ping 방지. 단일 매개변수로 안전한 우회 제공
  • 왜 fail-soft (|| { 경고 }): ping 실패해도 sitemap 자체는 정상 생성됐으니 빌드 결과 버리지 X. 사용자가 수동 재시도 안내

⑤ robots.txt 주석 추가

  • 한 일: key location URL을 주석으로 명시 (3줄)
  • 왜: 일부 third-party 크롤러·SEO 도구가 robots.txt에서 IndexNow key 위치를 hint로 활용. 기능 영향 0이지만 관행 준수가 안전
  • 왜 주석으로: 표준 robots.txt 디렉티브가 아니라 비표준 hint라 # 처리. 미래 봇이 인식 못해도 문제 X

⑥ Dry-run 검증

  • 한 일: bash scripts/indexnow-ping.sh --dry-run 실행 → payload 출력 → 4가지 항목 확인
  • 확인 항목:
    • ✓ shell syntax (bash -n)
    • ✓ URL 67개 추출 (sitemap과 일치)
    • ✓ JSON payload 형식 valid
    • ✓ image/upscale/(noindex 봉인) 자동 제외
    • ✓ tools/storage/(신규 도구) 정상 포함
  • 왜 비가역 작업 전 필수: 잘못된 host·잘못된 key·URL host 불일치(422) 등은 dry-run 없이는 실 전송에서야 발견. ping rate limit(429) 한계 상 디버깅 사이클 비쌈

⑦ Deploy + 실 전송 (deploy 의존성)

  • 한 일: commit + push → GitHub Pages 배포 → {key}.txt 공개 → ping 실행
  • 왜 deploy 후에야 가능: IndexNow API가 ping 받자마자 keyLocation GET해서 검증. deploy 안 된 상태에서 ping 보내면 403 Forbidden. 따라서 commit+push가 사실상 ping의 전제조건
  • 왜 ping 후 1~2분 wait 권장: GitHub Pages CDN 전파에 짧은 lead time. 너무 빨리 ping 하면 일부 IndexNow 검증 노드가 캐시 X 응답 받음

⑧ 효과 가시화 (1~3일 후)

  • 해야 할 일: Bing Webmaster Tools 가입 → "Submitted via IndexNow" 카운터 확인 / Naver Search Advisor에서 신규 URL 색인 속도 비교 / site:taystudio.github.io 검색 결과 수 변화
  • 왜 직접 인덱스 카운트는 못 봄: IndexNow API 자체는 ping 응답에 인덱스 결과 안 줌. 큐 진입만 보고 = 사후 검색엔진별 도구로 검증
  • 왜 1~3일 wait: ping → 크롤 → 인덱스 큐 → 인덱스 노출까지 단계별 lead time. 즉시 검증 X. 기대값 셋업이 중요 (사용자가 1시간 후 보고 "효과 없네" 결론 X)
학습 — 정적 사이트 워크플로우의 정신 이번 적용 = 새 인프라(서버·CI·DB) 0 추가하고 기존 빌드 스크립트에 7줄 + 새 파일 2개로 끝. plan.md §0.2의 "정적 한계까지 끌어올린다" 정신 그대로. push-style 프로토콜이라도 정적 사이트로 cover 가능한 케이스의 모범 사례.

9. 핵심 정리

  • 본질 = HTTP 한 줄(POST + GET 검증)로 검색엔진 인덱스 큐에 push
  • 정적 사이트 친화 = 백엔드 0, key 파일 1개 + 스크립트 1개
  • 한국 시장 효과 = 네이버 합류 + Bing 시리즈 + Yandex 동시 cover
  • 한계 = Google 미지원 + 인덱스 자체는 보장 X (속도만 가속)
