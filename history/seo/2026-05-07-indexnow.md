# IndexNow 인프라 도입 — 67 URL 일괄 push (2026-05-07)

> Search Console 일일 색인 요청 한도(~10건/일) 우회 + 한국 시장(Naver) 색인 가속 목적. **Bing·Yandex·Naver(2024 합류)·Seznam** 동시 전파. Google 미지원이지만 sitemap·SC로 cover 중. 백엔드 0 — 정적 사이트에 key 파일 1개 + bash 스크립트 1개로 끝.

## 결정 배경

- Search Console URL 검사 → 색인 요청은 **하루 ~10건 제한** (구글 정책). 67 URL 일괄 색인 안 됨
- 네이버 Search Advisor sitemap 제출은 **pull 방식** = 며칠 lead time
- IndexNow = **push 방식** + 단일 endpoint(api.indexnow.org)로 4개 엔진 동시 cover. Naver 2024년 합류로 한국 시장 ROI 결정적
- 정적 사이트 친화 = 인증 없이 key 파일 검증만, 백엔드 0

## 구현 결정

| 항목 | 채택 | 이유 |
|---|---|---|
| **endpoint** | `https://api.indexnow.org/indexnow` (Bing hub) | 단일 POST로 4 엔진 relay |
| **key 발급** | UUID 32자 lowercase hex (`b762f70e...`) | 충돌 0, 스펙 준수(8~128자) |
| **key 위치** | 루트 직속 `{key}.txt` | IndexNow 관행 + 일부 엔진 자동 추측 |
| **스크립트 분리** | `indexnow-ping.sh` standalone + `build-sitemap.sh` hook | 단일 책임 + 재시도 가능 + opt-out |
| **default 동작** | hook ON, `INDEXNOW_PING=0` opt-out | 까먹음 방지 + commit 전 우회 |
| **dry-run 옵션** | `--dry-run` 플래그 | 비가역 작업 사전 검증 |
| **fail 처리** | fail-soft (`||` 경고) | sitemap은 정상 생성됨, ping 실패가 빌드 망치지 X |

## 적용 단계 회고 (개념서 §8 정리)

상세 = `history/seo/concept-indexnow.html` §8 적용 단계별 작업 + 의미. 8 단계:

1. 사전 조사 (build-sitemap.sh·sitemap·robots 구조 파악)
2. Key 발급 + `{key}.txt` 루트 배치
3. Ping 스크립트(`indexnow-ping.sh`) 작성 + dry-run 옵션
4. 빌드 스크립트에 hook (default ON + opt-out)
5. robots.txt 주석 추가 (관행)
6. Dry-run 검증 (4 항목)
7. commit + push → deploy (~1~2분) → 실 ping
8. 효과 가시화 (Bing WT·Naver Advisor — 1~3일)

## Bing Webmaster Tools 가입 진행 (2026-05-07)

IndexNow 효과 가시화 1순위 채널. 소유권 검증 코드 발급받음 → meta 태그 적용.

### 적용 결정

| 항목 | 채택 | 이유 |
|---|---|---|
| 검증 방식 | **meta 태그** | 기존 `google-site-verification`·`naver-site-verification` 패턴과 일관 |
| XML 파일 방식 | 미적용 | meta로 충분, 추가 파일 노이즈 회피. 만약 검증 실패 시 fallback으로 추가 |
| DNS CNAME 방식 | 불가 | `taystudio.github.io` DNS 제어 X (GitHub 관리) |
| 적용 페이지 | 루트 `index.html`만 | Bing이 도메인 root만 fetch해서 검증 → 도메인 전체 권한 부여. 다른 페이지엔 노이즈만 됨 |

### 검증 흐름

## 학습 포인트

1. **정적 사이트의 push-protocol cover 가능성 검증** — sitemap = pull 모델 = 정적 친화. IndexNow = push 모델 = 보통 백엔드 필요. 그러나 **CLI 1회 실행 = "필요할 때 push 한 번 보내는" cron-less ping**으로 정적 사이트도 push 패턴 cover 가능. `plan.md §0.2 정적 한계까지 끌어올린다` 정신 구체 사례
2. **단일 진입점 통합 가치** — sitemap 빌드와 ping을 하나의 명령으로 묶으면 사용자 mental model 단순화. opt-out 옵션으로 안전한 우회 보장
3. **비가역 작업 = dry-run 필수** — IndexNow ping은 취소 X + rate limit 있음. 첫 실 전송 전 dry-run = 시간·비용 0의 안전망
4. **fail-soft 디자인** — sitemap 빌드와 ping은 책임이 다름. ping 실패가 sitemap 결과를 무효화하면 안 됨 = `||` 경고 처리. 후속 책임은 사용자 수동 재시도
5. **개념 문서 = HTML 우선** — markdown(seo/concepts.md)도 좋지만, 다이어그램·표·callout이 풍부한 문서는 HTML이 가독성 압승. site-chrome.css 재사용으로 작성 비용 낮음
6. **Naver IndexNow 합류(2024)는 한국 시장 결정적** — 그 전엔 한국 cover가 없어 IndexNow 도입 ROI 약했음. 2024 이후 = 한국 단일 사이트라도 도입 ROI 큼
