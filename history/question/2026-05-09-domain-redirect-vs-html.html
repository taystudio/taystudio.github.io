TAYSTUDIO · history · question · 도메인 redirect vs HTML 갱신

Q. 301 redirect로 옛 도메인 들어와도 새 도메인으로 가는데, HTML은 왜 다 바꿔야 하나?

2026-05-09 작성 / 카테고리: SEO · 도메인 migration / 관련: history/seo/2026-05-09-domain-migration.md
QUESTION

"DNS·CNAME 다 등록해서 옛 taystudio.github.io 들어와도 새 taystudios.com으로 redirect 잘 되는데, 그러면 HTML 안의 taystudio.github.io 표기는 안 바꿔도 되는 거 아닌가?"

한 줄 정리

  • 동작 측면: 맞음. 301 redirect로 사용자 도착은 정상
  • SEO 측면: 틀림. canonical 태그 하나만으로도 권위 이전이 깨짐
  • 핵심 원리: 301 = "이사 갔다", canonical = "원본은 여기다". 둘이 다른 곳을 가리키면 검색엔진이 혼란
  • 결론: HTML 일괄 갱신은 canonical 하나 때문이라도 필수. OG·JSON-LD·sitemap은 같이 가는 정합성 작업

1. 동작 vs SEO 신호 — 분리해서 보기

동작 (사용자 경험)

GitHub Pages가 custom domain 설정되면 옛 *.github.io URL을 자동 301 redirect함. 사용자·크롤러가 옛 URL을 GET 하면:

[사용자]                  [GitHub Pages 옛 도메인]            [새 도메인]
   │                              │                                │
1. │ GET taystudio.github.io/foo/                                  │
   ├─────────────────────────────>│                                │
   │                              │                                │
2. │       HTTP 301 Moved         │                                │
   │       Location: taystudios.com/foo/                           │
   │<─────────────────────────────┤                                │
   │                              │                                │
3. │ GET taystudios.com/foo/                                       │
   ├──────────────────────────────────────────────────────────────>│
   │                              │                                │
4. │       HTTP 200 OK + HTML                                      │
   │<──────────────────────────────────────────────────────────────┤

여기까지는 사용자 입장에서 "옛 URL 입력 → 새 도메인으로 잘 도착". 동작 끝.

SEO 신호 (검색엔진 크롤러)

그런데 4번 단계의 HTML 응답 안에 다음과 같은 메타가 박혀 있다고 하자 (HTML 갱신 안 한 상태):

<link rel="canonical" href="https://taystudio.github.io/foo/">
<meta property="og:url" content="https://taystudio.github.io/foo/">
<script type="application/ld+json">
{
  "@type": "WebApplication",
  "url": "https://taystudio.github.io/foo/"
}
</script>
핵심 모순 "당신은 새 도메인으로 도착했지만, 이 페이지의 원본은 옛 도메인입니다." — 페이지가 자기 자신을 부정하는 상태.

크롤러는 두 신호를 동시에 받음:

  1. 301 redirect: "원본은 새 도메인이다"
  2. canonical 태그: "원본은 옛 도메인이다"

이게 SEO 신호 충돌(conflicting canonical). 검색엔진은 둘 중 하나를 골라야 함.

2. 신호별 영향 — 안 바꾸면 무슨 일이 생기나

HTML 안의 신호안 바꾸면 일어나는 일심각도
<link rel="canonical"> 새 도메인 페이지가 "원본은 옛 도메인이다" 자기 부인 → 검색엔진이 옛 URL을 canonical로 인덱싱 → 301과 충돌하는 신호 → 권위 이전 몇 주~수개월 지연 → 검색 결과 등락폭 큼 치명
og:url · og:image:secure_url 카톡·페이스북·X 공유 카드 텍스트에 옛 도메인 노출. 클릭하면 301로 새 도메인 가지만 카드 자체가 옛 브랜드 = 사용자 혼란 + 브랜드 신뢰 손상 중간
JSON-LD url · @id · WebApplication · Organization 구조화 데이터의 모든 entity가 옛 도메인 식별자에 묶임 → Knowledge Panel·rich result가 옛 도메인 기준으로 누적 → 새 도메인 권위 0부터 다시 시작 위험 중간
sitemap.xml <loc> 새 도메인 Search Console에 옛 URL sitemap 제출 = property mismatch 경고. IndexNow는 옛 URL 67건 ping → 각 URL이 301 한 번 더 거쳐서 비효율 + queueing 효과 ↓ 중간
내부 <a href="https://taystudio.github.io/..."> 사용자가 사이트 내 링크 클릭할 때마다 301 한 번 더. 동작은 함. 단 페이지 로드 시간 +100~300ms 약함
hreflang(영어 분기 진입 시) 한↔영 페이지 alternate 표기가 옛 도메인이면 두 언어 페이지가 서로 못 찾음 = hreflang 자체가 작동 X 중간

3. canonical이 가장 치명인 이유 — 깊은 분석

나머지 신호는 "안 바꿔도 동작은 함, 효율만 떨어짐"이지만 canonical은 검색엔진이 어떤 URL을 인덱싱·랭킹할지 결정하는 1차 신호. 충돌 시 SEO 결과가 비결정적으로 변함.

Google이 충돌하는 canonical을 만나면

Google 공식 가이드(Consolidate duplicate URLs)에 따르면 다음을 종합 판단:

  1. 301 redirect 신호 (강함)
  2. rel="canonical" 태그 (강함)
  3. sitemap에 명시된 URL
  4. 내부·외부 링크가 가리키는 URL (백링크)
  5. HTTPS 우선·짧은 URL 선호

이상적인 케이스는 1~5번이 모두 같은 URL을 가리킴. 그래야 Google이 단일 canonical을 빠르게 확정 → 권위 이전·인덱싱 안정.

1번(301)과 2번(canonical)이 다르면 = "강한 두 신호 충돌". 보통 301이 이기지만:

  • 인덱스 결정 시간이 길어짐 — 며칠~몇 주가 몇 주~몇 달로 늘어남
  • SERP 등락폭 ↑ — 일시적으로 옛 URL이 노출되거나 둘 다 안 나오는 구간 발생
  • 일부 페이지는 옛 URL이 canonical로 확정되는 케이스 — 그 페이지는 새 도메인 권위 못 받음

왜 1~3개월 자연 deindex가 정상인가

모든 신호가 일치된 상태에서도 옛 도메인 자연 deindex는 1~3개월 걸림. 신호 충돌이 있으면 이게 6개월~1년으로 늘어날 수 있고, 그 동안:

  • 사용자가 SERP에서 클릭 → 옛 URL → 301 → 새 도메인 (3-hop). UX 손상 누적
  • 새 도메인 권위 자체가 약하게 누적 → AdSense RPM·트래픽 모두 손실
  • 옛 URL과 새 URL이 동시 존재 → "duplicate content" 페널티 위험 (드물지만)

4. 그럼 안 바꿔도 되는 건 뭐가 있나

도메인 migration에서 의도적으로 옛 표기를 보존한 곳:

위치이유
common/site-chrome.js ALLOWED 리스트의 옛 도메인 두 줄 전환기 6개월~1년 동안 옛 URL 직접 GET 시 mirror 경고 false positive 방지. ALLOWED 리스트는 클라이언트 가드용이라 SEO 신호 X — 검색엔진이 안 봄
history/seo/* 일자별 옛 entry 시점 기록의 정합성 보존. 갱신하면 "그 날의 사실관계"가 흐려짐. 시점 분기 entry(2026-05-09-domain-migration.md)로 새 정책 명시
history/migration/index.html 본문 옛→새 transition 자체를 기록하는 문서. "옛 도메인" 표기가 의미 있음
dash-tay9k3m/index.html GA·repo URL 안내 GA4 속성명·실제 GitHub repo 경로는 도메인과 별개. 사용자 결정 영역

= "검색엔진이 신호로 읽는 곳"은 모두 새 도메인. "내부 운영 도구·역사 기록"은 옛 표기 보존."

5. 정리 — 한 장으로

핵심 한 줄 301은 "이사 갔다"는 신호. canonical은 "원본은 여기다"라는 신호. 도메인 migration의 정석 = 모든 신호를 새 도메인으로 일치시킨다 + 301로 이전 명시. 둘 중 하나만 하면 권위 이전 효율 폭락.

작업 우선순위 (만약 시간 없을 때)

  1. canonical — 무조건 1순위. 한 곳도 빠지면 안 됨
  2. sitemap.xml — Search Console·IndexNow가 이걸 source of truth로 삼음
  3. JSON-LD url·@id — Knowledge Panel·rich result 권위
  4. og:url·og:image:secure_url — SNS 공유 브랜드 일관성
  5. 내부 <a> 절대 URL — 사용자 경험·페이지 로드 속도

실제 작업은 sed 한 번에 다 잡는 게 단순·안전 (검수 비용 최소화). 우리 케이스는 1~5 모두 단일 commit으로 일괄 적용.
